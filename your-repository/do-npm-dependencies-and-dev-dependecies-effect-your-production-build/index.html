Do NPM Dependencies and DevDependencies Affect Your Production Build?

Technical Guide • JavaScript Ecosystem • By Matthew Burfield

When managing a Node.js project, the distinction between dependencies and devDependencies in your package.json is one of the most fundamental yet misunderstood concepts. Does it actually matter where you save a package?

The Short Answer

In a properly configured production build (like Webpack, Vite, or Rollup), devDependencies do not impact the final bundle size sent to the user. However, they do impact your server-side environment and build pipeline speed.

Understanding the Difference

The npm install --production command is the key differentiator here. When you run this on a server:

  • Dependencies: Are installed because they are required for the application to run (e.g., Express, React, Lodash).
  • DevDependencies: Are skipped. These are only needed for local development (e.g., Jest, ESLint, TypeScript compilers).
// Example package.json structure
{
  "dependencies": {
    "express": "^4.17.1"
  },
  "devDependencies": {
    "nodemon": "^2.0.7",
    "jest": "^26.6.3"
  }
}

Why It Matters for Production

If you mistakenly put a large build tool in your dependencies, your production server will waste disk space and memory downloading it, even if it's never used by the running code. Conversely, if you put a runtime library in devDependencies, your app will crash in production because the library won't be found.

Pro Tip: Always use npm prune --production in your CI/CD pipeline to ensure no "dev" weight accidentally makes it into your production environment.
